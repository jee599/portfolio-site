3세션, 336 tool calls, 수정 파일 12개. 구현한 것보다 결정한 것이 더 많은 하루였다.

**TL;DR** 영어 기본 언어 전환은 단순해 보이지만 전체 레이아웃의 기본 텍스트를 전부 뒤집는 작업이었다. Claude Code가 파일을 읽고 패턴을 분석해서 변경 범위를 먼저 파악해줬기 때문에 삽질 없이 한 번에 끝났다.

## 세션 1: "영어 기본이어야 해" — 13 Read, 12 Edit

프롬프트는 단순했다.

```
저기서 지금 번역버튼이 잘 안되는데, 일단 영어 사이트가 기본이여서
영어 번역이 모든 부분에 대해서 되어 있어야해.
```

Claude Code가 먼저 현재 번역 시스템을 분석했다. `Base.astro`를 읽고 나서 문제를 3개로 정리했다.

1. `var lang = 'ko'`로 기본 언어가 하드코딩됨
2. 버튼 텍스트가 `EN`으로 표시되고 있음 (영어 기본이면 `KO`여야 함)
3. 모든 `data-ko`/`data-en` 요소의 기본 텍스트가 한국어

세 번째가 핵심이었다. 단순히 `lang = 'en'`으로 바꾸면 안 된다. HTML에 렌더링된 기본 텍스트 자체가 한국어라서, `data-en` 속성에서 가져오는 텍스트로 교체하거나 기본 텍스트를 영어로 바꿔야 한다.

Claude Code가 제안한 전략: `data-ko`/`data-en` 속성 구조는 그대로 두고, HTML에 적힌 기본 텍스트를 영어 버전으로 교체. JS 초기화 시 `localStorage`에서 `lang` 값을 읽어 한국어 사용자한테는 `data-ko`로 덮어씌우는 방식이다.

50 tool calls 중 25가 Read/Edit이었다. 나머지 절반이 Agent로 번역 검수를 위임한 것이다.

```
모든 영어 번역잘 되는지, 번역 매끄러운지 오타 없는지 찾아봐
```

이 프롬프트 하나로 Agent가 전체 사이트를 순회하며 번역 누락과 어색한 표현을 체크했다. 내가 수동으로 클릭해가며 확인하는 것보다 훨씬 빠르다.

## 세션 2: auto-publish 스킬에서 네이버 빼기 — 37 Bash, 23 Edit

원래 목적은 Claude 최신 업데이트 관련 글 3개를 DEV.to와 Hashnode에 발행하는 것이었다. 그런데 스킬을 열어보니 네이버가 발행 대상에 포함되어 있었다.

```
그 스킬에서 네이버꺼는 뺴줘
```

`auto-publish` 스킬에서 네이버 관련 부분을 전부 제거했다. Agent 3(네이버 한국어 HTML), `naver-seo-rules.md` 레퍼런스, Phase 4 네이버 발행 섹션, Phase 5 네이버 큐 확인. 6개 레퍼런스가 5개로 줄고, 발행 플랫폼이 3개에서 2개(spoonai.me, DEV.to)로 정리됐다.

이후에 Hashnode 토큰 설정이 생각보다 번거로웠다. 토큰을 직접 알려줬는데도 Bash로 환경변수를 찾고, `publish-to-hashnode.mjs` 스크립트를 수정하고, 다시 확인하는 과정이 37 Bash calls로 이어졌다. 토큰 하나 설정에 이렇게 많은 쉘 명령이 필요하다는 게 의외였다.

SEO 관점에서 핵심 설정이 하나 있다. DEV.to와 Hashnode에 올라가는 모든 영어 글의 `canonical_url`을 `jidonglab.com` 기준으로 잡는 것이다. 구글이 어느 플랫폼을 먼저 인덱싱하든, 원본 출처는 jidonglab.com으로 귀속된다. 중복 콘텐츠 패널티 없이 두 플랫폼에 동시 발행할 수 있는 이유다.

## 세션 3: 호버 애니메이션과 보안 감사 — 100 Bash, 33 Edit

가장 긴 세션이었다. 196 tool calls, 그 중 Bash가 100번.

시작은 refmade 사이트의 카드 호버 효과였다.

```
미리보기들이 움직이게 하거나 할 수 있어? 마우스 호버시에 미리보기
스크롤이나 애니메이션이 다시 실행된다던지
```

`ArticleCard.tsx`에 3가지 애니메이션을 추가했다. Hero 카드는 이미지 1.05x 줌인 + shimmer 스캔라인, Default 카드는 이미지 1.05x 줌인, Compact 카드는 썸네일 1.1x 줌인. `globals.css`에 scan 키프레임 애니메이션도 추가했다.

그런데 Vercel 배포가 계속 CANCELED됐다. `git push` 후 자동 배포가 세 번 연속으로 취소되는 상황. Claude Code가 수동으로 `vercel build --prod && vercel deploy --prebuilt --prod`로 우회해서 해결했다.

보안 감사는 프롬프트 하나로 Agent에 위임했다.

```
지금 이 사이트에 보안 이슈나 api 공격이나 탈취될만한 부분이 있어?
```

Agent가 돌아온 결과: CRITICAL 1건 수정. API 라우트에서 입력 검증이 없는 부분이었다. 이걸 내가 직접 찾으려면 코드를 처음부터 읽어야 했을 텐데, Agent 위임으로 15분 안에 끝났다.

호버 애니메이션은 요구사항이 계속 바뀌었다. "호버 시 스크롤" → "4초마다 스크롤 왜 돼?" → "호버할 때만 스크롤" → "스크롤 속도 2.5배" → "지금의 반". 요구사항이 구체화되는 과정을 Claude Code가 반복 수정으로 따라갔다. 33 Edit calls가 대부분 여기서 나왔다.

## 도구 사용 패턴

3세션 합산 도구별 분포다. Bash 149번, Edit 68번, Read 54번, Agent 25번, 나머지 40번. Bash가 절반에 가까운 건 빌드 확인, 배포, 환경변수 설정이 반복됐기 때문이다.

Agent 25번이 눈에 띈다. 번역 검수, 보안 감사, 레퍼런스 HTML 고도화를 전부 Agent로 위임했다. 내가 직접 하면 각각 1-2시간이 필요한 작업들이다.

## 이번 세션에서 배운 것

언어 전환처럼 단순해 보이는 작업도 Claude Code가 변경 범위를 먼저 분석하게 하면 사이드이펙트를 미리 잡을 수 있다. "영어로 바꿔줘"가 아니라 "현재 번역 시스템이 어떻게 동작하는지 파악하고, 문제를 정리해줘"가 더 나은 프롬프트다.

보안 감사와 코드 리뷰는 Agent 위임이 기본이 됐다. 직접 읽는 것보다 빠르고, 뭘 찾아야 할지 내가 미리 알 필요도 없다.
