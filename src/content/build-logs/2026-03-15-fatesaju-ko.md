세션 21개, tool call 총합 1,300+, 가장 긴 세션 하나에서만 515번. 사주 앱 fatesaju를 Claude Code로 처음부터 끝까지 만들었다. 코드 한 줄 직접 안 썼다.

이 빌드로그는 코드 구현이 아니라 Claude Code를 어떻게 썼는지에 관한 기록이다.

## 세션 1: 7개 배치, 76 tool calls, 8시간 16분

첫 세션 프롬프트가 길었다. 이메일 로그인, DB 스키마, 결제, 리포트 생성, 공유 기능까지 스펙 전체를 한 번에 던졌다.

Claude는 코드베이스 탐색 후 `Batch 1`부터 시작했다. DB 스키마, Supabase 클라이언트, 타입 정의. `Batch 2`는 결제 라이브러리, Auth 헬퍼, API 라우트. 이런 식으로 7개 배치를 순서대로 실행하면서 Write 37번, Read 18번, Bash 18번. 세션 하나에서 파일 37개가 생성됐다.

여기서 배운 게 있다. 스펙을 작게 쪼개서 여러 세션에 나눠 던지는 것보다, 전체 맥락을 한 번에 주면 Claude가 더 일관성 있게 짠다. 세션 간 컨텍스트가 끊기면 코드 스타일이 섞이고 타입 불일치가 생긴다.

## 디자인 반복: 3개 세션, 색깔 하나 고치는 데

세션 3에서 색깔 피드백을 줬다. "느낌이 촌스럽고 색감 통일성이 없다." 이걸 받아서 Claude가 `globals.css`, `theme.ts`, 컴포넌트 20개를 한꺼번에 수정했다. Read 24번, Write 23번.

근데 세션 도중 이런 에러가 떴다.

```
API Error: Claude's response exceeded the 32000 output token maximum.
```

출력 토큰 한도를 넘긴 것이다. 파일을 너무 많이 한 번에 쓰려다가 중간에 잘렸다. 이후엔 대형 작업을 할 때 `CLAUDE_CODE_MAX_OUTPUT_TOKENS` 환경변수를 올려두거나, 배치 크기를 의식적으로 나눴다.

세션 4에서 "프리미엄 느낌, 글로벌 SaaS"로 방향을 더 구체화했다. 여기서 DateInput 컴포넌트도 함께 새로 만들었다. Edit 16번, Write 13번. 세션 5에서는 또 DateInput을 달력 팝업으로 교체했다. 같은 컴포넌트를 3개 세션에 걸쳐 완성한 셈이다.

디자인은 한 번에 완성되지 않는다. 짧은 피드백 루프를 반복해야 하는데, Claude Code가 그 루프를 빠르게 돌려주는 역할을 한다.

## Plan Mode: 먼저 계획하고, 승인받고, 실행

세션 8에서 처음 Plan Mode를 썼다. 무료/유료 티어 분리 + 토큰 비용 추적 구현 전에 Claude가 `~/.claude/plans/glittery-herding-porcupine.md`에 계획을 작성했다. EnterPlanMode 1번, Read 11번, Write 1번.

계획 파일에 컨텍스트, 목표, 구현 단계가 정리됐고, 세션 9에서 그 계획을 실행했다. Edit 9번, Write 6번, TaskUpdate 16번.

큰 작업을 할수록 이 패턴이 효과적이었다. 계획 없이 바로 구현에 들어가면 중간에 방향이 틀어지거나 파일 간 의존성이 꼬인다. Plan Mode로 먼저 합의하고 실행하면 번복이 줄어든다.

세션 15에서도 같은 패턴을 썼다. Ops 대시보드(`/log`) 구현 전에 `vast-tumbling-mountain.md` 계획 파일을 먼저 작성하고, 세션 16에서 Edit 20번, Write 16번으로 실행했다.

## 멀티 AI 협업: WORK_LOG.md

세션 21은 특이했다. Claude Code + Anthropic Gemini가 같은 레포에서 번갈아 작업하는 구조를 설계했다. 동시에 작업하는 게 아니라, 한 AI가 끝나면 다음 AI가 이어받는 방식이다.

이를 위해 `WORK_LOG.md`를 만들었다. 초기 기획, 구현 상황, 남은 작업, 구현 목표, 지침. 각 AI는 작업 시작 전 이 파일을 읽고, 끝나면 업데이트한다.

파일에 이런 내용이 들어갔다.

```
여러 AI에서 사용하는 레포다. 작업 전 반드시 이 파일을 읽어라.
작업 후 반드시 업데이트해라.
QA 돌리고, 커밋 푸시하고, 로그 꼼꼼히 남겨라.
```

AI마다 어떤 작업을 했는지 추적할 수 있게 AI별 작업 로그도 별도 섹션에 기록했다. 세션이 길어지거나 컨텍스트가 초기화될 때 이 파일이 새 세션의 출발점이 됐다.

## QA 실패와 사주 검증

세션 21에서 가장 오래 걸린 게 QA였다. 실제 생년월일을 입력했더니 사주팔자가 틀렸다.

```
1995-04-28 · 11:18 · 여
```

에이전트 4명을 띄워서 검증을 시작했다. 뭐가 틀렸는지, 원래 값이 뭔지, 어디서 계산이 잘못됐는지. 시주 처리, 절기 기준, 음력/양력 변환 로직을 전부 뜯어봤다.

세션 11에서도 엔진 고도화 작업이 있었다. 격국, 천간합충, 파해 감지를 추가하면서 골든 테스트 139개를 작성했다. Edit 46번, Bash 21번. 테스트 없이 명리 계산 엔진을 신뢰하기 어렵다. 수학적으로 검증 가능한 부분은 반드시 단위 테스트로 고정했다.

LLM 비용 최적화도 Claude Code로 했다. 세션 14에서 시스템 프롬프트를 1,200 토큰으로 압축하고 Anthropic Prompt Caching을 적용했다. 같은 입력이 반복될 때 캐시 히트가 발생하면서 비용이 줄었다. 이 작업도 Read 9번, Edit 파일 5개.
