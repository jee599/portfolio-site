세션 하나에 423번의 도구 호출이 나왔다.

73시간 53분. 그 결과물은 jidonglab.com의 정체성 자체가 바뀌는 것이었다. AI 뉴스 사이트에서 **프로젝트 포트폴리오 허브**로.


## 왜 바꾸기로 했는가

jidonglab.com은 처음에 AI 뉴스 큐레이션 사이트로 시작했다. 매일 9시에 뉴스를 자동으로 긁어와 요약하고, DEV.to로 발행하는 파이프라인이 돌아갔다.

문제는 이게 포트폴리오 역할을 하지 못한다는 거였다.

로컬에 11개의 git 프로젝트가 있다. 사주 앱, 커피챗, 치과 사이트, 트레이딩 봇, AI 뉴스 플랫폼. 그런데 포트폴리오에는 7개만 등록되어 있고, 빌드 로그는 수동으로 쓰고 있었다. 뭔가 보여줄 게 있는데 어디에도 제대로 정리되어 있지 않은 상태.

그래서 방향을 전환했다. 사이트의 핵심을 뉴스에서 프로젝트로 바꾸는 것.

프롬프트는 이렇게 시작했다.

```
Implement the following plan:
jidonglab을 AI 뉴스/블로그 사이트가 아니라 프로젝트 포트폴리오 허브로 전환한다.
로컬에 11개 git 프로젝트가 있지만 포트폴리오에는 7개만 등록되어 있고,
빌드 로그는 수동 생성이다.
```

계획서 형식으로 넣었다. Claude Code는 계획서를 받으면 파일 단위로 실행 순서를 정하고 병렬 처리 가능한 것들은 묶어서 처리한다. "이것 좀 해줘"보다 훨씬 빠르다.


## admin.astro 58KB 파일과 맞닥뜨렸을 때

가장 먼저 막힌 건 `admin.astro`였다.

파일 크기가 58KB. Astro 컴포넌트 하나가 58KB면 뭔가 잘못된 거다. Projects, Build Logs, Stats, Dev.to 등 여러 탭의 UI와 로직이 전부 한 파일에 몰려 있었다.

Claude Code는 이 파일을 통째로 읽고 구조를 파악한 다음, 새 탭을 어디에 끼워넣어야 하는지 판단했다. 수동으로 했으면 스크롤만 30분이었을 텐데, Read 도구로 파일을 청크 단위로 읽으며 탭 구조와 JavaScript 이벤트 바인딩 위치를 파악했다.

여기서 배운 게 있다. "파일 수정해줘"보다 "이 파일의 탭 구조를 파악하고, Projects 탭과 Build Logs 탭을 추가해줘. 기존 JS 이벤트 바인딩 패턴을 유지해서"라고 구체적으로 지시하면 훨씬 정확하게 작동한다. 파일이 크고 복잡할수록 맥락을 더 많이 줘야 한다.


## GitHub API 연동: 9개 → 28개

관리자 페이지에서 프로젝트를 직접 등록하는 대신, GitHub 레포지토리를 긁어와서 보여주는 방식으로 바꾸기로 했다.

처음에는 잘 됐다. API 엔드포인트(`admin-projects.ts`)를 만들고, GitHub API를 호출해서 레포지토리 목록을 가져오는 로직을 붙였다. 그러다 `403 Forbidden` 에러가 났다.

토큰 권한 문제였다. GitHub Personal Access Token에 `repo` 권한이 필요한데 없었다. Claude Code가 이걸 직접 해결해줄 수는 없다. 토큰은 내가 직접 생성해야 한다. 여기서 잠깐 멈췄다.

권한을 다시 설정하고 토큰을 넣었더니 "9 registered, 28 on GitHub"가 떴다. 9개에서 28개로. GitHub에 올려둔 모든 레포지토리가 한 번에 잡혔다.

이걸로 뭘 할 수 있느냐. Admin에서 각 레포지토리의 `visible` 상태를 토글하면 포트폴리오 페이지에서 보이고 안 보이고를 바꿀 수 있다. 수동으로 YAML 파일을 수정하지 않아도 된다. 이게 핵심이었다.


## 빌드 로그 자동화 파이프라인

이 사이트에서 가장 오래 걸리던 작업이 빌드 로그 작성이었다. 뭔가 만들면 그걸 정리해서 글로 쓰는 데 추가로 시간이 들었다. 코딩보다 글쓰기가 더 오래 걸릴 때도 있었다.

`scripts/generate-build-log.sh`와 `scripts/parse-sessions.py`를 만들었다.

Claude Code의 `.jsonl` 세션 로그를 파싱해서 빌드 로그의 원재료를 추출하는 스크립트다. 어떤 파일을 수정했는지, 어떤 프롬프트를 입력했는지, 도구 호출은 몇 번 했는지. 이 데이터를 Claude에게 넘기면 빌드 로그 초안이 나온다.

그리고 `~/Library/LaunchAgents/com.jidong.build-log.plist`를 등록했다. 6시간마다 자동으로 세션 로그를 파싱하고 업데이트한다.

완전 자동은 아니다. 초안은 자동으로 나오지만 발행은 내가 검토하고 누른다. 글의 품질을 유지하려면 사람이 한 번은 봐야 한다. 지금은 자동 생성 + 수동 발행 구조다.


## 에이전트 병렬 실행이 실제로 유효한 경우

agentochester 프로젝트(세션 7, 327 tool calls)에서 `subagent-driven-development` 스킬을 썼다. 명세서를 받아서 독립적인 태스크로 쪼개고, 각 태스크를 별도 에이전트에게 위임하는 방식이다.

실제로 돌아간 에이전트만 나열하면: `adapter.ts` 구현, builtin YAML 에이전트 8개 생성, `catalog + agent-manager` 구현, `assembler.ts` 구현, TypeScript type safety 체크, Security audit, Test coverage 분석, Dashboard QA, Spec compliance 리뷰. 9개 에이전트가 각각 독립된 컨텍스트에서 돌았다.

단, 에이전트를 많이 쓴다고 무조건 빠른 건 아니다. 태스크 사이에 의존성이 있으면 순차 실행이 강제된다. "`adapter.ts`가 완성되어야 `catalog.ts`를 만들 수 있다"는 의존 관계가 있으면 병렬로 띄워도 결국 앞 태스크를 기다려야 한다. 명세서를 쓸 때 의존성 관계를 명확히 표시해두는 게 병렬 실행의 전제 조건이다.


## Dev.to 한글 포스트 정리와 SEO 작업

작업 중에 한 가지 문제를 발견했다.

Dev.to에 한국어로 된 글이 올라가 있었다. 영어 플랫폼에 한글 글은 노출도 안 되고 검색도 안 된다.

```
devto에 한글로 된 거 다 내려주고
지금 올라간 기능들에 구글검색 잘되게 하거나, 후킹, 유입이나 광고가 잘되게 세팅되어 있는지 확인해줘
```

Claude Code는 Dev.to API를 통해 현재 올라가 있는 글 목록을 가져오고, 언어를 판별해서 한글 글을 unpublish 처리했다. 그리고 남은 영어 글의 제목과 태그를 SEO 관점에서 재작성했다.

SEO의 핵심은 단순하다. 검색 의도를 파악하고, 제목에 핵심 키워드를 앞에 배치하고, 태그 4개를 전략적으로 선택하는 것. Claude Code가 이걸 기존 글 전체에 일괄 적용했다.


## admin YAML 에러와 에지 케이스

```
admin에서 링크나 project 상태 붙이면 yaml 어쩌고 에러메세지 나와
```

Admin UI에서 URL을 입력했더니 YAML 파싱이 깨졌다. URL에 포함된 `:` 때문이었다. YAML에서 콜론은 키-값 구분자다. 따옴표 없이 URL을 값으로 넣으면 파서가 혼란스러워한다.

처리는 간단하다. 문자열 값을 작성할 때 자동으로 따옴표로 감싸는 로직을 추가하면 된다. 문제는 이런 에지 케이스가 직접 써보기 전까지는 보이지 않는다는 점이다.

Claude Code로 기능을 빠르게 만들고, 실제로 쓰면서 깨지는 부분을 바로 고치는 사이클. 이게 1인 개발에서 가장 실용적인 방식이다. 완벽하게 만들려고 미리 모든 케이스를 설계하는 것보다, 일단 만들고 쓰면서 고치는 게 빠르다.


## 도구 사용 통계가 말해주는 것

Session 1 기준: `Bash(248), Edit(72), Read(63), Write(19), Grep(10)`

Bash가 248번으로 가장 많다. 배포, 빌드 확인, API 테스트, git 작업이 전부 Bash다. Edit이 72번. 기존 파일을 부분 수정하는 게 전체 파일 교체보다 훨씬 많다. Read가 63번. 큰 파일을 이해하려면 읽어야 한다. Write는 19번. 새 파일을 만드는 건 생각보다 적다.

이 비율이 올바른 작업 방식이다. 기존 코드를 이해하고 → 필요한 부분만 수정하고 → 새 파일은 꼭 필요할 때만 만든다.

Session 7(agentochester): `Bash(107), Edit(59), Read(49), Write(43), TaskUpdate(29)`

여기서 Write가 43번으로 높다. 새 레포지토리에서 처음부터 만드는 작업이라 새 파일 생성이 많았다. `TaskUpdate` 29번은 서브에이전트 태스크 상태를 업데이트하는 호출이다. 에이전트 오케스트레이션의 오버헤드가 숫자로 보인다.


## 지금 시점에서 남은 것

자동화가 붙었지만 아직 완전하지 않다.

빌드 로그 초안은 자동으로 생성되지만 퀄리티는 내가 넣은 재료에 달렸다. 세션 중에 어떤 프롬프트를 썼는지, 어떤 삽질이 있었는지가 파싱 가능한 형태로 남아야 한다. 대충 작업하면 초안도 대충 나온다.

GitHub 레포 28개가 잡혔지만 전부 포트폴리오에 넣을 건 아니다. `visible` 토글을 하나씩 확인하면서 정리해야 한다. 그리고 projects 페이지 리디자인. 어떤 프로젝트가 진행 중이고 어떤 게 완료된 건지 한 눈에 보이게 바꿀 예정이다.

> 사이트가 뭘 보여주는지 명확해지면, 거기서 일하는 방식도 달라진다.
