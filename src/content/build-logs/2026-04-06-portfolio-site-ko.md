48시간 30분, 670 tool calls. 단일 세션 기준 역대 최대다. 목표는 하나였다 — 치과 블로그 이미지 25장을 S급 수준으로 자동 생성하는 파이프라인을 만드는 것.

**TL;DR**: 서브에이전트 3개를 병렬 교차검증에 투입하고, Gemini Pro/Flash를 이미지 품질에 따라 분기했다. 4 라운드를 돌아야 AI 티 9점이 나왔다. 스킬 파일이 살아있는 문서가 돼야 파이프라인이 진화한다.

## "spoonai" 한 단어 프롬프트가 만든 38분 세션

두 번째 세션 전에 짧은 세션이 하나 있었다. 프롬프트는 한 단어였다.

```
spoonai
```

Claude는 메모리에서 두 개의 관련 프로젝트를 찾고 뭘 원하는지 물었다. 이후 추가 지시가 들어왔다.

```
번역기능 완벽하게 되도록 수정해줘
```

번역 버그의 원인은 `lib/content.ts:133`에 있었다. `getPostSlugs()`가 `!f.includes("-en")`으로 영어 파일을 전부 필터링하고 있었다. `getAllPosts("en")`이 빈 배열을 반환하니 `/posts/..-en` URL이 전부 404. 번역 UI는 돌아가는데 페이지가 없었던 것이다.

38분, 133 tool calls. Edit 63회, Bash 37회, Read 27회.

## 670 tool calls의 48시간: 치과 블로그 파이프라인

두 번째 세션이 본편이다. 네이버 S급 치과 블로그 3편(임플란트 뼈이식, 잇몸질환, 소아치과)을 위한 자동 이미지 생성 파이프라인. 각 포스트당 25장. 텍스트 카드 6장 + 3D 의료 일러스트 6장 + 실사진 활용 카드.

### "별도의 컨텍스트로 검증해줘" 패턴

이 세션의 핵심 프롬프트는 이것이었다.

```
래퍼런스 s급들이랑 지금 너가 만든 결과랑 일치하는지
모든 방면에서 별도의 컨텍스트 기준으로 평가하고
더 수정하거나 보완할 점 이야기해줘
```

Claude는 이걸 Agent 도구 3개 동시 실행으로 해석했다. 라운드마다 3개 에이전트를 병렬로 투입했다. 역할은 고정이었다.

- SEO/알고리즘 검증 에이전트
- 의료 정확성 + 의료광고법 검증 에이전트
- S급 디자인/비주얼 기준 검증 에이전트

같은 파일 3편을 3개 컨텍스트가 동시에 읽고 각기 다른 기준으로 채점한다. 메인 컨텍스트가 "좋아 보인다"고 판단한 것들이 각 검증 에이전트에서 6~7점을 받았다. 생성한 컨텍스트가 직접 검증하면 편향이 생긴다. 별도 컨텍스트에서 결과물만 보면 독자 관점이 더 가깝다.

### Gemini Pro/Flash 라우팅

초기엔 모든 이미지를 같은 모델로 생성했다. 라운드 2에서 지시가 명확해졌다.

```
pro랑 flash 필요한 그림 퀄리티에 따라 다르게 써줘
```

분기 기준은 이미지 타입이었다. 의료 일러스트(해부학, 치조골 단면, 임플란트 식립 과정) → Gemini 2.0 Pro. 텍스트 카드(히어로, FAQ, CTA, 면책고지) → Gemini 2.0 Flash. Pro는 품질이 높지만 느리고 비싸다. 카드형 이미지에 Pro를 쓰는 건 낭비다.

`pipeline.py`의 `build_template_variables()`에 파일명 패턴으로 타입을 판단하고 모델을 선택하는 로직이 들어갔다.

### 라운드 1→4: AI 티 9점까지

라운드 1의 가장 큰 문제는 3D CGI 스타일이었다. 파란 글로우, 과장된 광택, 로봇 같은 질감. AI 티 점수 평균 6~7점.

라운드 2에서 `viewport` 버그를 잡았다. HTML 템플릿의 `body height: 860px` 고정인데 hero 캡처 clip height가 1075라서 히어로 이미지 하단 215px이 흰 배경으로 나왔다. 텍스트 카드 로고 이중 노출 버그도 수정했다.

라운드 3에서 의료 일러스트 스타일을 Gray's Anatomy 교과서 스타일로 전환했다. 3D CGI → 단순화된 해부학 일러스트. AI 티 점수가 8~9점으로 올라갔다.

라운드 4는 최종 튜닝이었다.

```
ai 티 9점이상이여야할 것 같아
```

에이전트 3개가 같은 25장을 독립적으로 채점했다. 점수가 수렴하면 통과, 벌어지면 가장 낮은 항목 파악 후 수정. 최종 결과물: 25장 평균 AI 티 점수 9.1점.

### 스킬 파일이 살아있는 문서가 돼야 한다

이 세션에서 `naver-dental-blog/SKILL.md`와 `dental-blog-image-pipeline/SKILL.md`가 7번 업데이트됐다.

```
지금까지 작업 스킬 업데이트하고, 남은 작업 뭐야? 진행해
```

이 패턴이 반복됐다. 스킬 업데이트를 "나중에"로 미루면 다음 세션에서 처음부터 다시 학습한다. 세션 내에서 발견한 S급 기준, 실패한 프롬프트 패턴, 버그 수정 사항을 그 자리에서 스킬에 적어야 파이프라인이 누적된다.

라운드를 거치며 `pipeline.py`에 추가된 것들:

- `parse_image_tags()` — `<!-- IMAGE: -->` 태그 파서
- `build_template_variables()` — 파일명 기반 Pro/Flash 분기
- CTA 카드 차별화 — 같은 CTA 템플릿이 반복 사용되지 않도록 파일명 기반 분기
- `viewport` 고정 → HTML 템플릿 전체에 `overflow: hidden` 적용

## 삽질: 롤백이 불가능한 순간

라운드 5 근처에서 전면 리디자인이 들어갔다.

```
[Image] 이거는 바로 직전버전이 더 나은데? 롤백해
```

에이전트가 파일을 직접 수정했기 때문에 git history가 없었다. 백업 없이 전면 리디자인을 돌리면 이전 버전으로 돌아갈 수 없다. 이 세션의 가장 비싼 실수다.

파이프라인이 파일을 덮어쓰기 전에 `.bak` 백업을 만들거나, 각 라운드 결과를 별도 디렉토리에 저장하는 로직이 필요하다. 에이전트를 이용한 반복 작업에서는 git commit을 라운드 단위로 찍어두는 게 맞다.

삽질이 하나 더 있었다. 파이프라인에 재시도 루프가 3개여서 성공해도 3번 실행됐다. 사용자가 "1차 2차 3차가 왜 있어?"라고 물어서야 발견했다. 1회 성공하면 종료, 실패 시에만 재시도로 수정했다.

## 670 tool calls의 분포

Bash가 가장 많다. `pipeline.py` 실행, 이미지 생성, 파일 확인이 반복됐다. Agent는 20회 이상. 3개 병렬 검증, 개별 블로그 수정, 재검증 사이클. Edit은 상대적으로 적다. 코드 수정보다 파이프라인 실행과 검증이 대부분이었다.

48시간 세션이지만 실제 활성 작업 시간은 훨씬 짧다. 이미지 생성이 배치로 돌아가는 동안 대기 시간이 포함됐다. `run_in_background`로 파이프라인을 백그라운드에 던지고 검증 에이전트를 동시에 돌리는 방식이 전체 흐름을 빠르게 했다.

에이전트 병렬 실행에서 중요한 건 스코프 분리다. 검증 에이전트들이 읽기만 하면 파일 충돌이 없다. 수정 에이전트가 들어올 때는 각자 다른 파일을 담당하게 해야 한다. 이 세션에서는 검증/수정 에이전트를 교대로 썼기 때문에 충돌이 없었다.
