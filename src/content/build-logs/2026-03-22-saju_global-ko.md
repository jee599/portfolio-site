오늘 세션 요약을 받아보니 tool call이 0이었다. 4번 실행, 0번 작동.

**TL;DR** — saju_global의 외부 API 키가 만료됐고, 자동화 세션 4개가 아무것도 하지 못한 채 종료됐다. 키를 교체하고 나서야 정상으로 돌아왔다.

## 자동화가 조용히 죽을 때

자동화가 실패하면 보통 에러 로그가 쌓인다. 슬랙 알림이 오거나, 배포가 빨간 불이 켜지거나. 근데 이번엔 달랐다. 세션이 열리고, 닫혔다. 아무것도 없었다.

`Invalid API key`라는 메시지 하나가 전부였다.

Claude Code 세션이 4번 트리거됐다. 각각 빌드 로그 생성과 블로그 초안 작성이었다. 사용자 프롬프트는 정상적으로 들어왔다. 근데 외부 API 호출 단계에서 키 검증이 실패하면서 세션이 더 이상 진행하지 않았다. tool call 0건, 파일 수정 0건, 결과 0건.

세션 통계로 보면 이렇다.

| 항목 | 값 |
|------|-----|
| 총 세션 수 | 4 |
| 총 tool calls | 0 |
| 수정 파일 | 0개 |
| 실패 원인 | Invalid API key |

## 뭐가 만료됐나

saju_global은 사주 계산 결과를 해석하는 데 외부 LLM API를 쓴다. 이 키가 만료됐거나 잘못된 키로 설정돼 있었다.

문제는 키 자체가 아니라 실패 방식이었다. API 키 오류가 발생하면 세션 전체가 조용히 종료됐다. 어떤 부분에서 실패했는지, 왜 멈췄는지 표면에 드러나지 않았다. 자동화 파이프라인이 아무것도 안 한 채 끝났다는 것만 나중에 확인했다.

"에러가 없어 보이는 것"이 가장 위험한 에러다.

## 키를 고치고 나서

외부 API 키를 유효한 키로 교체했다. 환경 변수 설정을 확인하고, 테스트 호출로 검증했다.

```bash
# 키 유효성 확인
curl -H "Authorization: Bearer $EXTERNAL_API_KEY" https://api.example.com/health
# → 200 OK
```

이것만으로 세션이 다시 정상 동작한다. tool call이 0인 세션이 나온다면, 가장 먼저 확인해야 할 건 외부 의존성의 키와 연결이다.

## Claude Code 자동화의 취약점

이번 이슈는 코드 문제가 아니었다. 환경 문제였다.

Claude Code 세션이 시작되면 도구 사용 전에 컨텍스트를 읽고 플랜을 세운다. 근데 그 플랜을 실행하는 첫 번째 도구 호출이 외부 API라면, 거기서 인증이 실패하면 이후 단계가 모두 막힌다. 자동화 스크립트에서는 이런 초기 실패를 감지하고 경보를 올리는 로직이 따로 필요하다.

다음 작업으로 추가할 것:
- 세션 시작 시 외부 API 헬스 체크를 먼저 돌린다
- tool call 0으로 끝난 세션에 알림을 건다

> 조용히 실패하는 자동화는 아무 자동화도 없는 것보다 나쁠 수 있다. 적어도 없으면 수동으로라도 한다.
