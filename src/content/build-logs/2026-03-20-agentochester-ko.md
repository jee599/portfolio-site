"피카츄 배구 멀티로 만들어줘."

이게 전부였다. 6글자짜리 프롬프트 하나로 1시간 53분이 시작됐다.

**TL;DR** agentochester로 에이전트 4명을 병렬 디스패치해서 피카츄 배구 멀티플레이어 게임을 만들었다. 첫 결과물은 "장난해?" 소리를 들었고, 원본 소스를 찾아 물리 엔진을 다시 구현하는 과정까지 총 197 tool calls이 들었다.


## "장난해?" — 첫 번째 결과물의 문제

에이전트 3명을 동시에 보냈다. `game_designer`는 게임 메커닉 문서를 쓰고, `frontend_developer`는 Next.js + Canvas 클라이언트를 만들고, `backend_architect`는 WebSocket 서버를 구축한다. 세 명이 동시에 작업하니까 순차적으로 했을 때보다 훨씬 빠르게 뼈대가 나온다.

`qa_engineer`까지 4명이 완료됐을 때 결과는 이랬다. 빌드 성공. 테스트 92개 통과. 서버 타입체크 통과.

그런데 실제로 켜보니 문제가 있었다.

"리소스랑 게임의 느낌이 원본 피카츄 게임이 아니잖아."

당연한 얘기다. 에이전트들이 만든 건 피카츄 테마를 가진 배구 게임이지, 원본 피카츄 배구가 아니었다. 1997년에 나온 그 게임 특유의 물리 엔진, 스프라이트, 조작감이 없었다.

이게 병렬 에이전트 디스패치의 함정이다. 빠르게 만들어주지만, 뭘 만들지 공유된 맥락이 없으면 방향이 각자 달라진다. `game_designer`는 일반적인 배구 규칙 문서를 썼고, `frontend_developer`는 그걸 보고 그냥 배구 게임을 만들었다.


## 원본 소스를 찾아서

여기서 접근을 바꿨다. 에이전트한테 "원본 피카츄 배구 소스를 찾아서 리소스를 가져와"라고 했다.

원본 피카츄 배구는 오픈소스로 공개되어 있다. 에이전트가 소스를 분석해서 핵심 물리 로직을 추출했다. 공의 초기 y 좌표, 스파이크 판정 방식, 캐릭터 점프 곡선 같은 것들이 원본이랑 정확히 달랐다.

`Rewrite server - original physics` 에이전트와 `Rewrite client - sprites + physics` 에이전트를 다시 병렬로 보냈다. TypeScript strict mode 통과, 빌드 성공.

그다음엔 코드를 별도 레포로 분리했다. `~/agentochester/pikachu-volleyball/`에 있던 파일들을 `git@github.com:jee599/pikachu.git`으로 옮겼다. "워크스페이스 하나 만들고 피카츄 배구 관련 코드 모두 옮겨줘"라고 하니까 에이전트가 알아서 처리했다.


## 조작 버그 — 원본 로직 역공학

게임이 돌아가기 시작하니까 세부 버그가 나왔다. 공중에서 다이빙이 됐고, 엔터 키가 스파이크로 작동하지 않았고, 60프레임이 안 나왔다.

"원본 소스에서 로직 없어? 점프한 상태에서 다이빙이 안 되고, 엔터랑 방향키 동시에 누르면 그쪽으로 스파이크 되게 하는 로직."

이 질문이 핵심이었다. 에이전트가 원본 소스를 전부 읽고 비교 결과를 정리해줬다. 우리 코드 vs 원본 코드 비교표를 보니까 어디가 다른지 바로 보였다. 원본은 방향키 8방향으로 스파이크가 구현되어 있었고, 점프 상태 플래그가 별도로 관리됐다.

"모든 소스 다 확인해서 제대로 구현해"라는 지시 하나로 에이전트가 원본 로직을 역공학해서 `engine.ts`, `input.ts`, `game.ts` 세 파일을 수정했다.


## 197 tool calls의 분포

이 세션에서 사용된 도구 호출을 보면 작업 패턴이 읽힌다.

Bash 84회, Read 48회, Edit 36회, Write 17회, Agent 12회.

Bash가 가장 많은 건 빌드 확인, 서버 실행, 프로세스 정리 때문이다. 포트 3001이 이전 프로세스에 점유되어 있어서 정리하는 데만 몇 번이 들어갔다. Read가 48회인 건 원본 소스 분석 때 파일을 많이 읽었기 때문이다. Edit 36회는 세부 버그 수정이 그만큼 반복됐다는 뜻이다.

Agent가 12회인 건 생각보다 적어 보이지만, 에이전트 한 명이 내부적으로 수십 번의 도구 호출을 한다. 12번의 `Agent` 호출 뒤에 197번 전체가 숨어 있다.


## agentochester가 하는 일

이 프로젝트의 본래 목적은 에이전트 디스패치 CLI다. "피카츄 배구 멀티로 만들어줘" 같은 프롬프트를 받으면 작업을 분해해서 적합한 에이전트 역할에 배분한다.

세션 1~3에서 봤던 것처럼, 태스크 디컴포저가 프롬프트를 받아서 JSON으로 역할과 액션을 내뱉는다.

```json
[
  {"role": "frontend_developer", "action": "Next.js + Tailwind CSS 기반 블로그 사이트 구현"},
  {"role": "seo_specialist", "action": "메타태그, Open Graph, sitemap.xml 최적화"},
  {"role": "ui_designer", "action": "레이아웃, 타이포그래피, 반응형 디자인 검수"}
]
```

`frontend_developer`, `backend_architect`, `qa_engineer`, `security_engineer` 같은 역할이 정의되어 있고, 복잡한 요청이 들어오면 이 역할들로 분해된다.

세션 4에서 실제 게임 개발에 이걸 적용해봤다. 효과는 있었다. 병렬로 돌리니까 순차적 작업보다 확실히 빠르다. 다만 맥락 공유가 없으면 각 에이전트가 서로 다른 방향으로 구현할 수 있다는 걸 확인했다.

다음에 비슷한 작업을 할 때는 에이전트를 보내기 전에 맥락 문서를 먼저 만들어서 공유해야 한다. "원본 피카츄 배구와 동일한 물리 엔진"이라는 조건이 처음부터 있었다면 첫 번째 결과물이 달랐을 것이다.

> 에이전트를 병렬로 보내기 전에 공유할 맥락을 먼저 만들어야 한다. 빠른 게 능사가 아니다.
