"최악이야 결과물, 이전에 스킬 고도화 한 거 어디갔어?"

이 한 마디에 세션 2의 방향이 바뀌었다. 에이전트 5개를 병렬로 띄워 블로그 전체를 평가하고, 파이프라인을 처음부터 다시 잡았다. 3개 세션, 총 454 tool calls의 기록이다.

**TL;DR**: 이미지 파이프라인 고도화는 코드 수정보다 기준 정의가 먼저다. "어떤 결과가 나와야 하는지"를 에이전트에게 명확히 주지 않으면, 파이프라인을 열 번 돌려도 같은 문제가 반복된다.

## Cron이 2주 동안 죽어있었다

세션 1은 짧았다. `~/spoonai` 디렉토리에서 이메일 발송 문제를 추적했다. 마지막 아카이브가 2026-03-30, 오늘이 4/13 — 2주 넘게 스케줄러가 멈춰있었다.

원인은 단순했다. CronCreate로 등록할 때 `durable: true` 옵션을 빠뜨렸다. 세션이 종료되면 cron도 같이 죽는다. `crontab -l`도 비어있었고 launchd plist도 없었다. Cowork 태스크로 재등록하면 된다는 걸 확인하고 세션이 끝났다.

텔레그램 연결 문제도 이날 함께 나왔다. "텔레그램으로 메시지 보내면 답장 왜 안해"라는 질문. 이 세션이 텔레그램 채널에 연결돼 있지 않았고, `/telegram:configure`로 봇 토큰을 이 세션에 등록해야 수신이 된다. Bash 8회, Read 2회, Agent 1회로 세션이 끝났다.

## "최악이야" — 5개 에이전트 병렬 평가로 시작

세션 2가 본 작업이었다. uddental 치과 블로그 이미지 파이프라인 고도화. 261 tool calls, 25시간짜리 세션이다.

시작 프롬프트:

```
uddental 블로그 고도화 이어서
```

메모리에 미해결 3가지가 남아있었다. 이미지 퀄리티 근본 개선, 실사 사진 16장 분류, 블로그 글 전체 고도화. 001/002/003번 블로그 세 편이 있었고, 002(충치치료)가 이미지-텍스트 균형과 전문성 시그널 면에서 가장 잘 됐다는 걸 확인했다.

실사 사진 16장을 먼저 분류했다. Claude가 직접 보고 카테고리를 나눴다.

| 파일 | 카테고리 |
|------|---------|
| 37-001~003 | 외관 (건물 정면, 간판) |
| 45-001~004 | 복도/CT실 |
| 46-001~009 | 진료실, 대기실 |
| 47-001~014 | 접수, 각 진료실 |

`manifest.json`에 오분류가 있었다. 46-009(진료실 → 대기실), 45-004(복도 → CT실), 47-011 누락. 3개를 수정하고 CT실/대기실/안내 카테고리를 신설했다.

분류 완료 후 AgentCrow 패턴으로 3개 에이전트를 병렬 디스패치했다.

```
Dispatching 3 agents:
@skill-updater  → SKILL.md 사진 카테고리 7종 반영
@blog-enhancer  → 001 임플란트 블로그 글 전문성 고도화
@blog-reader    → 002/003 블로그 구조 분석 (기준점 도출)
```

## 결과물이 나오자마자 "최악이야"

파이프라인으로 이미지 24장을 생성하고 결과를 봤다. 반응이 즉각적이었다.

> "최악이야 결과물, 이전에 스킬 고도화 한 거 어디갔어?"

`naver-dental-blog` SKILL.md를 그대로 붙여줬다. 기준이 거기 있었다. S급 벤치마크 5곳, IMAGE 태그 22~25개, 다크 배경 카드 30%+, 글자수 5,000~6,000자.

스킬을 다시 읽고 001번 블로그를 기준에 맞게 재작성했다. 이번엔 에이전트가 검증 테이블까지 출력했다.

| 항목 | 기준 | 실제 | 충족 |
|------|------|------|------|
| 순수 텍스트 | 5,000~6,000자 | 6,016자 | O |
| IMAGE 태그 | 22~25개 | 24개 | O |
| H2 소제목 | 7~8개 | 8개 | O |
| FAQ | 5개+ | 7개 | O |

숫자는 맞았다. 문제는 이미지가 없다는 것. 파이프라인 재실행이 필요했다.

## 이미지 반복 조정 — 로고, 배경색, 한글, 분할

이미지 품질 조정이 세션의 절반을 차지했다. 스크린샷을 공유하고, 문제를 지적하고, 파이프라인을 수정하고 재실행하는 사이클을 반복했다.

**로고 사이즈**: 처음엔 너무 컸다. "지금 사이즈의 반~1/3 정도여야 할 것 같아"로 축소. 재실행 후 "지금 사이즈의 2배"로 되돌림. 로고 크기만 3번 반복했다.

**배경색**: 웜 베이지(`#f5f0eb`) → 약한 하늘색(`#f0f4ff`) → 순백(`#ffffff`). 매번 파이프라인을 재실행했다. 배경색 하나를 결정하는 데 에이전트 3번이 소비됐다.

**한글 필수**: 스크린샷에서 영어만 나온 이미지를 발견했다. "영어 없는 건 필수잖아"라는 피드백. `pipeline.py`의 Gemini 프롬프트에 한글 라벨 강제 규칙을 추가했다.

**이미지 분할**: "뭐 과정같은 거 보여줄 때 너무 한 장에 다 보여주는데." 임플란트 6단계가 3장(2단계씩)으로 묶여있었다. 1단계=1장으로 분리했다. 이미지 수가 늘었지만 정보 밀도가 낮아져서 읽기 쉬워졌다.

파이프라인 실패(`exit code 2`)도 여러 번 있었다. 세션 2에서만 8번 이상이다. 매번 에러 로그를 확인하고 즉시 다음 시도로 넘어갔다.

## 5개 에이전트 병렬로 전체 품질 평가

"전체적인 톤이 통일되지 않았다"는 피드백이 나왔다. 텍스트 톤, 디자인, SEO, 이미지 품질, 광고 효과 — 동시에 5개 관점이 필요한 상황이었다.

에이전트 5개를 병렬 디스패치했다.

```
Agent "블로그 톤/문체 평가"    → 어미 통일성 8/10, AI 클리셰 6건 발견
Agent "블로그 시각 디자인 평가" → 색상 통일성 6/10
Agent "블로그 콘텐츠/SEO 평가" → 키워드 밀도, 구조 분석
Agent "블로그 이미지 품질 평가" → 종합 8.2/10
Agent "블로그 광고효과 평가"   → CTA 효과 8/10
```

에이전트가 동시에 돌면서 각각 평가 리포트를 냈다. 색상 통일성이 6/10으로 가장 낮았다. 카드마다 배경색이 달랐다. 7개 템플릿 파일의 body background를 일괄 통일하고, `pipeline.py`에서 색상 팔레트를 하드코딩했다.

## SKILL.md가 코드와 함께 진화한다

`pipeline.py`를 수정할 때마다 `SKILL.md`도 함께 업데이트했다. 세션 2에서만 SKILL.md에 12개 Edit이 들어갔다.

이유는 명확하다. `pipeline.py`만 고치면 다음 세션 에이전트가 SKILL.md를 읽고 "이 스킬은 캡션 없음"으로 알고 원래대로 되돌릴 수 있다. SKILL.md가 소스 오브 트루스다.

```
pipeline.py 수정 → SKILL.md 동기화 → 다음 세션에서 일관성 유지
```

"수정한 내용 스킬에 반영해"라는 프롬프트 한 줄이 이 루프를 돌렸다.

## SEO 288 랜딩 페이지를 스크립트로 자동 생성

세션 3은 `saju_global` 프로젝트였다. SEO 궁합 랜딩 288페이지 구현.

```
saju_global 프로젝트에서 SEO 궁합 랜딩 288페이지를 구현해줘.
스펙: docs/superpowers/specs/2026-04-09-seo-compatibility-pages-design.md
```

`writing-plans` 스킬로 구현 계획을 먼저 작성했다. 스펙을 읽고, 기존 코드베이스의 패턴을 파악하고, 태스크별 파일을 정리한 뒤 플랜을 저장했다.

288페이지를 하나씩 만들 수는 없다. `generate-compat-content.ts` 스크립트를 짜서 백그라운드로 띄웠다. `zodiac-compat-content.json`에 각 궁합 조합의 콘텐츠를 생성하고 페이지 라우트를 자동으로 뿌리는 방식이다.

각 조합별 URL("양자리+천칭자리 궁합" 같은 롱테일 키워드)이 있으면 검색 노출 기회가 288배로 늘어난다. `subagent-driven-development` 스킬로 태스크별 에이전트를 디스패치하면서 git worktree를 별도로 만들어 main 브랜치를 건드리지 않고 작업했다.

세션 3 도구 사용: Bash 111회, Read 20회, Agent 17회, TaskUpdate 14회.

## 반복이 기준이 된다

이미지 조정은 삽질처럼 보이지만, 반복할수록 기준이 명확해진다. 로고가 너무 크면 얼마나 줄여야 하는지, 배경색이 웜 베이지면 왜 어색한지 — 이 판단이 쌓이면 `SKILL.md`의 기준이 된다.

다음 세션에서는 처음부터 "로고는 카드 너비의 15%, 배경 순백(`#ffffff`), 한글 라벨 필수"라고 명시한 스킬을 가지고 시작할 수 있다. 그게 이번 25시간 세션의 실질적인 산출물이다.
