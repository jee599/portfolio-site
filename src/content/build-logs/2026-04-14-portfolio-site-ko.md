2주 동안 스팸 필터 탓만 했다. 실제 원인은 Cowork cron이 조용히 죽어있었던 것.

**TL;DR** 세션 3개, 프로젝트 3개. 가장 큰 작업은 사주 앱 SEO 랜딩 288페이지 자동화, 가장 당황스러운 발견은 2주째 이메일을 못 보낸 spoonai 스케줄러였다.

## 토스페이먼츠가 공짜가 아니었다

첫 세션은 단순한 질문으로 시작했다.

```
지금 국내에서 결제 붙일 수 있는 방법 뭐뭐 있어? 무료
```

WebSearch 3번, Read 1번. Claude가 정리한 표에서 토스페이먼츠가 "초기비용 무료"로 나왔다. 그런데 추가 질문을 하니 실체가 드러났다.

> 맞다, 토스페이먼츠 직접 가입하면 비용이 든다. 가입비 22만원, 연 관리비 11만원, 카드 수수료 3.4%.

"무슨 등록비 30에 관리비 10?"이라고 물어봤는데, 실제로는 22만+11만이었다. 파트너사 경유 계약이거나 옵션이 붙은 케이스였던 것.

해결책은 포트원(PortOne) 경유. 포트원을 통해 PG사를 연동하면 가입비가 면제된다. 수수료만 내면 된다. 이 사실을 WebSearch로 3번 확인하고 나서야 확신이 생겼다. 리서치 목적 세션은 14 tool calls로 끝났다.

## 288페이지를 어떻게 만드나

두 번째 세션이 메인이었다. `saju_global` 프로젝트에서 SEO 궁합 랜딩 288페이지 구현.

프롬프트는 이것 하나였다.

```
saju_global 프로젝트에서 SEO 궁합 랜딩 288페이지를 구현해줘.
스펙: docs/superpowers/specs/2026-04-09-seo-compatibility-pages-design.md
```

`writing-plans` 스킬이 먼저 발동했다. Claude가 직접 코드베이스 전체를 탐색하고 구현 계획을 문서화했다. 완성된 플랜은 `docs/superpowers/plans/2026-04-10-seo-compatibility-pages.md`에 저장됐다.

그 다음 `subagent-driven-development` 스킬. 태스크별로 독립 에이전트를 디스패치한다. 각 에이전트는 자기 컨텍스트만 갖고 작업하고, 완료 후 스펙 준수 리뷰 + 코드 품질 리뷰가 돌아간다.

```
근데 그냥 cli써도 되잖아?
```

중간에 이런 질문이 나왔다. 서브에이전트 방식 대신 CLI에서 직접 스크립트 돌리는 방법이 있기 때문. 결국 Python 스크립트로 콘텐츠를 배치 생성하는 방향으로 실행했다. `nohup npx tsx scripts/generate-compat-content.ts` 백그라운드 실행, 로그 모니터링.

이 세션에서 Bash가 111번 호출됐다. 전체 207 tool calls 중 절반 이상이 여기에 몰렸다. 생성된 핵심 파일은 `zodiac-compat-content.json` — 궁합 콘텐츠 데이터를 담은 JSON이다.

워크트리 전략도 썼다. `main` 브랜치를 건드리지 않고 `.claude/worktrees/seo-compat-pages/` 에서 작업. 브랜치 오염 없이 대규모 파일 생성 가능.

## "잘 되고 있어?" 세 번

백그라운드 프로세스가 돌아가는 동안 "잘되고 있어?"를 두 번 물어봤다.

```
잘되고 있어?
잘되고 있어?
다됐어? 무슨 작업을 했어?
```

Claude는 그때마다 로그 파일을 읽어서 진행 상황을 요약했다. 이게 백그라운드 태스크 패턴의 실제 모습이다. 완료 알림이 오기 전까지 상태 확인은 수동이다.

세션 중반에 이메일 알림 설정 이야기도 나왔고, 도메인 통합(coffeechat → fortunelab) 이야기도 나왔다. 큰 세션에는 항상 이런 사이드 쿼리가 끼어든다. 182 tool calls, 92분짜리 세션이었다.

## Cron이 2주째 죽어 있었다

세 번째 세션.

```
구독자 리스트에 있는데 이메일 안가
```

이 메시지 하나로 시작했다. Bash로 로그 확인, 마지막 성공 발송이 2026-03-28. 오늘이 4월 13일이니까 정확히 16일이 침묵한 것.

원인 진단은 빨랐다.

```
crontab -l  # 비어있음
launchd plist  # 없음
```

`CronCreate`로 등록한 세션 cron이 `durable: true` 없이 만들어져서 세션 종료와 함께 소멸한 것이다. Cowork 태스크였는데 비활성화 상태였다.

해결 방법은 Cowork 대시보드에서 태스크를 다시 켜는 것. 프롬프트 칸에 `spoonai-daily-briefing 스킬 실행해줘`를 넣으면 전체 파이프라인 — 크롤링 → 기사 생성 → 이미지 → 배포 → 이메일 발송 — 이 다시 돌아간다.

교훈: cron을 등록할 때 `durable: true`를 빼먹으면 세션 종료와 함께 사라진다. 영속 스케줄러가 필요하면 반드시 옵션을 확인해야 한다.

## 세션 통계

| 세션 | 소요 시간 | tool calls | 주 작업 |
|------|---------|-----------|--------|
| 결제 리서치 | 39분 | 14 | PG 옵션 조사 |
| SEO 페이지 구현 | 92분 | 182 | 288페이지 자동화 |
| Cron 복구 | 13분 | 11 | 스케줄러 재등록 |

도구별로는 Bash 120회, Read 23회, Agent 19회, TaskUpdate 17회. Bash가 압도적으로 많은 건 스크립트 실행과 로그 확인이 반복됐기 때문이다. 파일 수정은 1개, 신규 생성은 2개뿐이었다. 대부분의 작업이 런타임에서 일어났다.

## 멀티 프로젝트 전환 비용

하루에 세 프로젝트를 오갔다. `spoonai`, `saju_global`, `spoonai` 다시. 각 세션은 `/clear`로 컨텍스트를 리셋하거나 새 세션으로 시작했다.

전환 비용은 생각보다 낮다. 프로젝트마다 `CLAUDE.md`가 있고, 첫 메시지로 현재 상태를 바로 파악한다. "spoonai 폴더 가줘"라고 하면 Claude가 디렉토리 구조를 읽고 컨텍스트를 잡는다.

문제는 장기 태스크다. 백그라운드에서 돌아가는 스크립트, 다른 세션에서 실행한 cron — 이것들은 세션이 바뀌면 상태 파악이 안 된다. 로그 파일이나 `STATUS.md` 같은 외부 상태 저장소가 필수다.
