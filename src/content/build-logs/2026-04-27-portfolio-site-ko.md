## AI GitHub 트렌드 3편, 어떻게 뽑았나

요청은 단순했다.

> "ai git에서 유명한 프로젝트 한 4개 정도 최신거 분석 글 devto에 블로그 올려줘"

`auto-publish` 스킬을 실행하면 Phase 1에서 WebSearch로 소재를 수집한다. 2026년 4월 기준 트렌딩 프로젝트로 `andrej-karpathy-skills` (Skills 패러다임 확산), `hermes-agent`, `OpenClaw` (로컬 AI 게이트웨이), `opencode` (터미널 에이전트) 네 개가 잡혔다.

4개를 3편으로 묶었다. 단순 목록이 아니라 테마별로 쪼갰다.

| 편 | 주제 | 다룬 프로젝트 |
|---|---|---|
| Part 1 | Skills 패러다임의 탄생 | karpathy/skills + hermes-agent |
| Part 2 | 로컬 AI 게이트웨이 전쟁 | OpenClaw |
| Part 3 | 터미널 에이전트 시대 | opencode |

시리즈명은 `The 2026 AI GitHub Playbook`. DEV.to에 같은 시리즈로 묶으면 이후 편이 올라갈 때 구독자에게 알림이 가는 구조를 이용했다.

Part 1은 즉시 발행, Parts 2-3은 `published: false`로 드래프트 업로드했다. 발행 타이밍을 쪼개기 위해서다. 스킬이 알아서 3편을 생성하고 DEV.to API로 순서대로 처리했다. 내가 한 건 "3편 정도", "진행"이라고 입력한 것뿐이다.

결과: Part 1 ["How a Markdown File Hit 16K Stars: Skills in 2026"](https://dev.to/ji_ai/how-a-markdown-file-hit-16k-stars-skills-in-2026-36hi) — DEV.to id=3542024, 2026-04-23 14:55 UTC 발행.

## spoonai 기사 품질: "26일자랑 25일자가 달라"

같은 파이프라인인데 날짜에 따라 기사 품질이 달랐다. 소스 기사의 깊이가 매일 다르면 생성 결과도 달라진다.

원하는 방향은 이랬다.

> "하나의 기사를 읽으면 거기에 관련된 내용 / 주변 상황이나 필요한 다른 지식까지 쉽게 습득할 수 있도록 + 사진이나 필요한 이미지를 삽입해서 읽히기 쉽게."

이걸 구현하기 위해 `self-critique.mjs`를 수정했다. 기사 초안을 생성한 뒤 같은 Claude API로 자체 비평을 한 번 더 돌리는 구조다.

```
~/spoonai-site/.claude/worktrees/quality-floor/scripts/lib/self-critique.mjs
```

self-critique 체크 항목은 세 가지였다. 본문 요약에 그치는지 혹은 관련 지식까지 아우르는지, 업계 인사이트가 포함됐는지, 이미지 삽입 위치가 텍스트 흐름을 방해하지 않는지.

Claude Haiku 기준으로 기사 한 편당 추가 비용은 $0.002 미만이다. 기사 품질을 기준 이하로 내보내는 비용이 훨씬 크다.

## 서브에이전트 12개 병렬: 치과 광고 시장 조사

세션 3은 완전히 다른 프로젝트였다. `~/dentalad/`로 이동해서 치과·병원 광고 시장 분석을 요청받았다.

> "최근 기준으로 치과 광고 어떤 식으로 돌아가는지 최근 5년 / 최근 1년 / 최근 3개월 트렌드 비교해줘. 분석해서 html로 딥하게 보고서 알려줘."

서브에이전트 12개를 병렬 디스패치했다. 각 에이전트에게 다른 도메인을 배정해서 중복 없이 수집하게 했다.

| 에이전트 | 담당 |
|---|---|
| 1-2 | 네이버 블로그·플레이스 알고리즘 최신 변화 (C-Rank, D.I.A.+) |
| 3-4 | 한국 AI 의료 광고 업체 지형도 (8개 카테고리 60곳) |
| 5-6 | Meta Advantage+, 구글 PMAX 의료 규제 현황 |
| 7-8 | 실제 AI 광고 산출물 사례 (블로그, 챗봇, 영상, 포트폴리오) |
| 9-10 | CPC 단가, ROI 수치 데이터 수집 |
| 11-12 | 의료광고법 위반 사례 + 규제 리스크 |

12개가 병렬로 돌아가는 동안 기다리는 시간은 없었다. 가장 느린 에이전트가 끝나면 결과를 취합해서 HTML 보고서를 작성했다.

생성된 파일 4개:
- `TREND-COMPARISON-REPORT.html` — 5년·1년·90일 비교표, 채널별 운영법
- `AI-AGENCIES-DEEP-REPORT.html` — 업체별 딥다이브 (가격·작동·클레임·리스크)
- `AI-AGENCIES-PRIMER.html` — 전문용어 없이 읽히는 입문서 버전
- `AI-AGENCIES-EXAMPLES.html` — 실제 산출물 갤러리

가장 값진 결과물은 갤러리였다. "AI 광고"를 표방하는 업체들이 실제로 어떤 결과물을 내는지를 한눈에 비교할 수 있었다. 실명+정량 수치가 있는 케이스(별 5개)는 2곳뿐이었다. 호원앤컴퍼니와 인블로그. 나머지는 이니셜이나 추정치로 메워졌다.

2시간 26분, 63 tool calls.

## Claude Code를 리서치 도구로 쓰는 패턴

이번 주 세션에서 반복된 구조가 있다.

**소재 수집 → 구조화 → 병렬 처리 → 통합 출력**

코드를 짜는 흐름과 같다. 결과물이 `.ts` 파일 대신 `.html` 보고서이거나 DEV.to 포스트일 뿐이다. 프롬프트를 구조화하고, 도메인을 쪼개서 병렬화하고, 결과를 취합하는 로직은 동일하다.

서브에이전트를 분할하는 기준은 단순하다. 서로 다른 URL을 봐야 하거나 서로 다른 검색 쿼리를 날려야 하는 작업은 분리한다. 같은 맥락을 공유해야만 하는 작업은 합친다.

12개를 던진다고 항상 좋은 건 아니다. 실제로 쓸 만한 자료를 가져온 에이전트는 절반이었다. 도메인을 더 좁게 쪼개는 편이 낫다는 걸 이번에 다시 확인했다.
