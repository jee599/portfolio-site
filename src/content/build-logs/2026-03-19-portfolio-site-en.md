423 tool calls. One session. 73 hours of wall-clock time.

Bash ran 248 times. Edit 72. Read 63. That's not a quick refactor or a feature add — that's a full identity change for a site.

This build log also happens to be the first post generated by the pipeline I built *during* the session, which makes the recursion feel appropriate.

**TL;DR:** I converted jidonglab.com from an AI news curation site to a project portfolio hub. I built a pipeline that parses Claude Code's JSONL session files to auto-generate build logs. I used multi-agent parallelism to batch-translate 6 existing posts into English simultaneously. And I fixed the quiet mess of Korean posts that had been publishing to DEV.to without a language filter.


## The Site Didn't Know What It Was

jidonglab.com started as an AI news site. Not by deliberate design — by accumulation.

The setup worked fine: a cron job fires at 9 AM and 9 PM KST, pulls recent AI/LLM news, Claude writes it up, and it publishes to DEV.to automatically. Consistent, running, no maintenance needed.

The problem: the site became *only* that.

I had 11 active projects locally. The portfolio showed 7. Build logs were written by hand when I remembered. And if you landed on jidonglab.com cold, you'd see walls of AI news summaries with no indication of what I actually build or why the site exists.

The core question — "what does this person make?" — went unanswered.

This is the kind of problem that's easy to tolerate while you're heads-down shipping. The site "works." There's content. But content that doesn't communicate anything about the person behind it isn't doing its job.

So I redefined it. jidonglab.com is a project portfolio hub. The news automation stays — it demonstrates the kind of systems I build — but it's a feature of the site, not the identity.

The implementation plan:

- `project-registry.yaml` as the single source of truth for project metadata
- GitHub API to pull live repo data
- Admin page with `visible` toggle to control what appears publicly
- Build logs auto-generated from Claude Code JSONL session files

Four lines. Several days to execute.


## Claude Code's JSONL Files Are a Gold Mine You're Not Using

Every Claude Code session gets stored locally in `.claude/projects/` as JSONL — one JSON object per line, local only, never uploaded anywhere.

Each line is one of three things: a user message, an assistant response, or a tool call and its result. Tool calls include the tool name, all inputs, and the complete output. Everything has a timestamp.

The practical implication: the full story of a development session — what you asked, what Claude ran, which files changed, what the outputs looked like — is sitting in a structured file on your disk.

I'd been writing build logs manually. After a long session, I'd reconstruct what happened, figure out which decisions mattered, and write it up. The JSONL already had all of that. I was doing redundant work.

`scripts/parse-sessions.py` fixes this. It processes the JSONL and extracts:

- User prompts (what I asked for)
- Tool call statistics broken down by type
- File paths that were read, edited, or created
- Session boundaries estimated from timestamp gaps

```python
# Session boundary: new session if gap > 30 minutes
if (current_timestamp - last_timestamp).seconds > 1800:
    start_new_session()

# Tool call counting
tool_stats[tool_name] = tool_stats.get(tool_name, 0) + 1

# File extraction from ToolResult outputs
if 'file_path' in result:
    changed_files.add(result['file_path'])
```

The parsed output feeds into `generate-build-log.sh`, which formats it and hands it to Claude: "Here's what happened in this session. Write a build log in the standard format."

Not fully hands-off — I edited the generated draft. But the 80% that's hardest to write from memory (what happened, in what order, what the actual numbers were) came out automatically.

**Where it breaks down:** The 30-minute gap heuristic for session boundaries is too coarse. Long sessions with breaks get split incorrectly. Short back-to-back sessions get merged. File path extraction has noise because different tools format their outputs differently. These are known gaps. The current state is "good enough to get to a publishable draft faster than writing from scratch," which is the bar that matters.


## GitHub API: Two 403s, Two Different Root Causes

GitHub integration was supposed to be the easy part.

The Admin page needs to show all my repos alongside registered projects — the goal is seeing the gap between "what's in the portfolio" and "what's on GitHub." Straightforward read operation.

**First 403:** Token scope. I issued a Personal Access Token with `public_repo` scope only. Reasonable for a portfolio, but I have private repos. They returned nothing. GitHub sends 403 (not 404) when you request resources outside your token's scope — a design choice that tells you the resource *exists* but is inaccessible. Re-issued the token with `repo` scope. All 28 repos appeared.

**Second 403:** Rate limiting. The Admin page hit the GitHub API on every page load during development — no caching, no deduplication. Rapid reloads during testing burned through the rate limit in minutes. Added response caching. Resolved.

The moment that felt like real progress: the Admin dashboard showed `(9 registered, 28 on GitHub)`. Two numbers, one line. 19 projects I hadn't surfaced in the portfolio yet. The gap was visible, controllable, and no longer required touching code to close.


## One Agent Dispatch, Six Translations, Twelve Files

The portfolio had 6 Korean build logs. DEV.to's audience is almost entirely English-speaking. The posts needed English versions.

The obvious approach: translate one at a time, review, move on. Fifteen to twenty minutes per post, assuming you're doing it carefully.

Instead: I dispatched a single agent with the task of translating all 6 simultaneously. One prompt, one dispatch. The agent ran the jobs in parallel and returned:

```
All 12 files created successfully.
```

6 Korean originals. 6 English rewrites. Done.

The instruction wasn't "translate" — it was "rewrite for English developers." Translation preserves the source text. Rewriting changes the audience assumption.

Korean-specific context — local service comparisons, Korean developer community references, anything that requires cultural background to land — got less emphasis. Technical content and Claude Code workflow patterns got more room. The framing shifted from "here's what I built" to "here's a pattern you can use."

This is where multi-agent starts to feel qualitatively different from a faster assistant. You're not just doing the same work more quickly. You're changing the shape of work — six parallel translation tasks go from "an afternoon block" to "background process while you're doing something else."

The other thing worth noting: the isolated context actually matters for quality. A translation task that runs in its own context window, focused on that one job, produces more consistent output than translation that happens as a side task during an active development session. Context hygiene is underrated.


## The Projects Tab: Three Iterations to Get It Right

The rebuild started with the schema. When I tried toggling project state from the Admin page, I got `yaml error`. The `visible` field existed in the YAML files but wasn't in the Content Collection schema in `src/content/config.ts`. Added the field to the schema, updated all existing YAML files.

Then came the UI work, which took three attempts.

**Version 1:** Text table. Project name, status, last updated. Clean, minimal. The problem: every project had identical visual weight regardless of state. "Live and generating revenue" and "paused indefinitely" looked the same. The requirement I was actually trying to satisfy — "I should be able to see what's active at a glance" — wasn't achievable with a uniform table.

**Version 2:** `ProjectCard.astro` component with color-coded status badges — green for Live, yellow for WIP, gray for Paused. Last build log date on each card. Project detail pages at `/projects/[slug]`.

Deployed. Opened the projects page. Every project was showing, including ones with `visible: false`.

The filter condition in `src/lib/projects.ts` wasn't applied. Fixed. Three iterations total.

This pattern is worth naming: three rounds of iteration sounds inefficient in planning terms, but each version was around 10 minutes with Claude Code. Three fast rounds cost less total time than one careful first attempt, and each round surfaced a problem (schema gap, filter bug, visual hierarchy issue) that couldn't have been spotted without building and seeing the thing.


## Filtering Korean Posts Out of DEV.to

The question that surfaced this: "Who posted Korean content to DEV.to?"

GitHub Actions. The `publish.yml` workflow synced everything to DEV.to without a language check. Korean build logs were publishing to an English-first platform, invisible to the intended audience and likely hurting engagement metrics.

Fix: added a language filter to the sync workflow.

```yaml
# .github/workflows/publish.yml
- name: Sync to DEV.to
  if: ${{ steps.check.outputs.lang != 'ko' }}
```

Korean posts excluded at the workflow level. Already-published Korean posts pulled via API.

At the same time, I ran a title pass on the existing English posts. The before/after is direct:

Before: `Building a Portfolio Site with Astro`

After: `How I Auto-Generate Build Logs from Claude Code JSONL`

The first title describes what the post is. The second answers "why should I read this?" A post title makes a promise to the reader. Specific results and numbers deliver on that promise before the click. Generic topics don't.


## Where Things Stand

The site's structure changed. jidonglab.com is now a project hub: 28 GitHub repos visible in the Admin panel, `visible` toggling for public display control, build logs generated from JSONL rather than written from memory.

Known gaps: JSONL parsing accuracy needs improvement (the session boundary heuristic is too rough for complex sessions), project detail pages are still minimal, Admin changes need faster revalidation to reflect immediately on the public site.

But the identity question is answered. If you land on jidonglab.com now, you can see what I build.
