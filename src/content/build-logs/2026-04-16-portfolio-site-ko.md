1h 39min. 16 tool call. 수정된 파일 0개. 그런데 이 세션이 쓸모없었다고 하기 어렵다.

**TL;DR** spoonai 프로젝트의 아카이브 이미지 미표시, 모바일 레이아웃 문제를 진단하는 세션이었다. 결론은 버그가 아니라 미구현. `ArchiveEntry` 타입에 `image` 필드 자체가 없었다.

## 잘못된 프롬프트로 출발

첫 프롬프트가 이거였다.

> "spoonai 프로젝트 가줘"

이어서:

> "디자인 전체적으로 수정해주고 모바일에서 제대로 안 보여"

이 프롬프트를 받은 Claude의 첫 반응은 되물음이었다. 모바일에서 구체적으로 뭐가 안 보이나. 헤더 메뉴 깨짐인지 카드 잘림인지 텍스트 오버플로인지. 디자인 방향은 다듬는 수준인지 리디자인인지. 우선순위 페이지는 어디인지.

이건 나쁜 반응이 아니다. "전체적으로 수정"은 범위가 너무 크다. 방향 없이 시작하면 나중에 더 많이 되돌린다. 세 번째 프롬프트에서 방향이 정해졌다.

> "라이브로 봐서 디자인 수정해주고 사진 아카이브에서 제대로 안나오는거랑, 모바일에서 제대로 안나와"

이 시점에서 전략이 바뀌었다. 라이브 사이트를 직접 보고, 아카이브 코드를 병렬로 파악한다.

## WebFetch로 라이브 사이트를 직접 본다

Bash 6번, Read 6번, WebFetch 2번. WebFetch 2번이 이 세션의 핵심이다.

라이브 사이트를 `WebFetch`로 직접 읽으면 코드만 봤을 때 놓치는 것들이 보인다. 어떤 이미지가 실제로 렌더링되고 있는지, 어떤 요소가 누락인지. 코드 분석과 라이브 확인을 병렬로 돌리면 원인 특정이 빠르다.

`Read` 6번 중 핵심은 두 파일이었다. `lib/types.ts`와 `ArchiveList.tsx`.

다만 `WebFetch`는 HTML을 텍스트로 변환할 뿐이라 실제 모바일 렌더링 레이아웃까지 확인하기는 어렵다. 모바일 버그 수정은 이 세션에서 완결되지 않았다.

## "버그"가 아니라 "미구현"이었다

아카이브 이미지가 안 나오는 문제의 루트 코즈는 명확했다.

`lib/types.ts`의 `ArchiveEntry` 타입에 `image` 필드 자체가 없다. `getArchiveEntries()` 함수는 포스트의 `meta.image`를 아예 버리고 `date`, `title`, `summary`만 담는다. `ArchiveList.tsx`는 텍스트 카드만 렌더링한다. 썸네일 자리가 없다.

```ts
// ArchiveEntry — image 필드 없음
type ArchiveEntry = {
  date: string;
  title: string;
  summary: string;
  // image?: string  ← 처음부터 없음
}
```

"사진이 안 나오는" 게 아니라 **아예 렌더링 안 하도록 구현된 것**이다. 이 차이는 수정 방향을 완전히 바꾼다. 버그 픽스가 아니라 기능 추가다. 타입 → 함수 → 컴포넌트 세 곳을 연쇄 수정해야 한다.

코드를 먼저 수정했다면 CSS나 레이아웃을 건드리며 엉뚱한 방향으로 삽질했을 가능성이 높다.

## 파일 0개 수정이 왜 가치있나

Claude Code 세션을 평가할 때 "얼마나 많이 바꿨나"로 보기 쉽다. tool call 수, 수정 파일 수, 커밋 수. 이 세션은 숫자가 다 낮다.

하지만 세션이 끝났을 때 두 가지를 알고 있었다. 아카이브 이미지 미표시는 타입 → 함수 → 컴포넌트 세 곳을 수정해야 한다. 모바일 문제의 구체적인 위치. 이 정보가 없으면 다음 세션에서 또 진단부터 시작한다.

> 어디를 수정해야 하는지 모르는 상태에서 코드를 건드리면 작업량만 늘어난다.

진단 세션의 가치는 다음 세션의 속도에 나타난다.
