Claude Code에서 에이전트를 쓰면 무조건 빠를 것 같지만, 실제 측정 결과는 다르다. 단순 작업에서 Teams를 쓰면 직접 처리보다 4배 이상 느렸다.

**TL;DR** AgentCrow v5는 요청을 분석해 직접/병렬/Teams 세 모드로 라우팅한다. 벤치마크 결과: 직접 41초 · 병렬 89초 · Teams 197초. 작업 성격에 따라 모드를 고르는 게 전부다.

## 계기: "에이전트 쓰면 Teams에 넘겨?"

대화는 이렇게 시작됐다.

```
나: agentcrow 적용됐어?
나: 에이전트 생성하면 알아서 agent teams에 넘겨?
Claude: 현재는 Teams 없이 직접 서브에이전트를 스폰합니다
나: 내가 원하는건 teams를 활용하는거야
나: teams랑 프롬프트 사이의 라우터 느낌으로 가고 싶어
```

AgentCrow는 이미 144개 에이전트 페르소나를 가진 라우터였는데, 실제로는 그냥 `Agent` 도구를 직접 호출하는 서브에이전트 패턴이었다. Teams의 `SendMessage` 핸드오프를 쓰지 않고 있었다. 둘의 차이가 구체적으로 얼마나 되는지 측정해보기로 했다.

## 벤치마크 설계

세 모드를 동일한 작업으로 비교했다. `slugify`, `deepClone`, `retry` 유틸 함수 구현 + 테스트 작성 — 독립적이고 크기가 비슷한 모듈 3개, 총 6개 파일.

**모드 1: 직접 처리** — 내가 순차로 6개 파일 직접 작성
**모드 2: 병렬 에이전트** — `Agent` 도구 3개 동시 호출, 각자 2개 파일 담당
**모드 3: Teams** — `TeamCreate` → `TaskCreate` → `Agent(team_name)` → `SendMessage` 핸드오프

의존성이 있는 케이스도 따로 돌렸다. `types` → `validator` → `formatter` 순으로 서로 import하는 v2 시나리오.

## 결과

| 모드 | 소요 시간 | Tool Calls |
|------|-----------|-----------|
| 직접 처리 | **41초** | 12 |
| 병렬 에이전트 | **89초** | ~45 |
| Teams | **197초** | ~80 |

Teams는 직접 처리보다 4.8배 느렸다. 스폰 오버헤드, `TeamCreate`, `TaskCreate`, `SendMessage` 왕복이 전부 시간을 잡아먹는다.

그런데 이게 Teams가 나쁘다는 뜻이 아니다. 의존성 있는 v2 시나리오에서는 답이 달랐다.

## 언제 무엇을 쓰나

**직접 처리**가 최선인 경우: 파일 1~5개, 스펙 명확, 단일 도메인. 에이전트 스폰 오버헤드 자체가 작업 시간보다 크면 손해다.

**병렬 에이전트**가 맞는 경우: 도메인 2개 이상, 파일 충돌 없음, 스펙을 프롬프트에 미리 쓸 수 있을 때. 의존성이 있어도 인터페이스를 텍스트로 전달 가능하면 병렬로 충분하다. v2 시나리오에서 `formatter`가 `types`를 import해야 했는데, 인터페이스를 프롬프트에 미리 써주면 굳이 Teams가 필요 없었다.

**Teams**가 필요한 경우: 에이전트 A의 **실행 결과**를 B가 받아야 할 때. DB 스키마 분석 후 마이그레이션 생성, 빌드 에러 수집 후 수정, 코드베이스 탐색 후 구현. "프롬프트에 미리 쓸 수 없는 동적 정보"가 핸드오프에 필요한 경우다.

## AgentCrow v5 판단 플로우

벤치마크 결과를 반영해서 CLAUDE.md를 업데이트했다.

```
Q1. 파일 5개 이하, 스펙 명확한가?
  YES → 직접 처리
  NO  ↓

Q2. 프롬프트에 각 에이전트가 필요한 정보를 다 쓸 수 있나?
  YES → 병렬 에이전트 (Agent 도구, Teams 없이)
  NO  ↓

Q3. 에이전트 A의 실행 결과를 B가 받아야 하나?
  YES → Teams (TeamCreate + SendMessage 핸드오프)
```

이전 v4는 "복합 요청이면 무조건 에이전트"로 단순화되어 있었다. 측정하고 나서야 세 경우를 구분해야 한다는 걸 알았다.

## 세션 병행: refmade 83개 레퍼런스 병렬 처리

같은 기간에 refmade 프로젝트에서 병렬 에이전트 패턴을 대규모로 썼다. Stripe, Vercel, Linear, Notion, Supabase, Clerk 등 실제 SaaS 랜딩 페이지를 HTML로 재현하는 작업, 남은 것만 83개.

프롬프트는 단순했다.

```
refmade 프로젝트 레퍼런스 구현→평가 루프 이어서 해줘. 007 재평가부터 시작
```

패턴은 명확했다. 4~5개씩 묶어서 병렬 스폰 → 각 에이전트가 원본 스크린샷과 비교해서 구현 → 평가 점수 9점 이상이면 PASS. 이걸 배치로 반복했다. 42시간짜리 세션에서 302 tool calls.

중간에 rate limit을 여러 번 맞았다. 에이전트가 `"You've hit your limit · resets 11pm (Asia/Seoul)"`을 반환하는데, 해당 에이전트만 멈추고 나머지는 계속 돌아간다. "다시 진행해" 프롬프트를 3번 쳤다.

이미지 품질 문제도 있었다. CSS 도형으로 만든 가짜 이미지들이 퀄리티를 망치고 있었는데, Gemini Imagen API로 해결했다. 각 레퍼런스에 맞는 이미지 프롬프트를 Claude가 직접 작성하고 API를 호출해서 HTML에 적용하는 흐름. `056-app-store`에는 auburn hair + cream blazer 여성, `064-neon-cinema`에는 실제 콘서트 무대, `073-poppr`에는 VR 전시 공간.

## 이번 세션 통계

- 총 세션: 5개
- 총 tool calls: 466
- 도구별: Read 196회, Bash 108회, Agent 75회, Write 19회
- 생성 파일: 17개, 수정 파일: 5개

## 배운 것

벤치마크를 직접 돌려보기 전까지는 "에이전트를 많이 쓸수록 좋다"고 막연히 생각했다. 측정하고 나서 기준이 생겼다. Teams가 필요한 순간은 생각보다 적다. 독립 작업은 병렬로 던지는 게 빠르고, 의존성이 있어도 정보를 프롬프트에 미리 쓸 수 있으면 Teams까지 갈 이유가 없다.

에이전트 오케스트레이션에서 중요한 건 "몇 명이냐"가 아니라 "어떤 모드로 어떻게 연결하느냐"다.
