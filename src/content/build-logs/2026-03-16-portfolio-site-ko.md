포트폴리오 사이트에 AI 뉴스가 매일 자동으로 올라오게 됐다. 그것도 DEV.to, Hashnode, Medium, 네이버 블로그까지 동시에. 세션 하나에서 324번의 도구 호출이 일어났다.

**TL;DR** "다 해줘" 한 마디에 Claude Code가 4개 플랫폼 자동화 파이프라인을 설계하고 구현했다. 단, 삽질 없이 된 건 하나도 없었다.

## "jidonglab이랑 devto 안되고 있으니까 될 때까지 고쳐"

이게 세션의 시작 프롬프트였다. 모호하다. 뭐가 안 되는지도 안 썼다.

근데 Claude Code는 알아서 파고든다. `src/lib/devto.ts` 읽고, GitHub Actions 워크플로우 파일 열고, 실제로 실패하고 있는 이유를 추적한다. "안 된다"는 말 한 마디로 진단부터 수정까지 간다.

이 세션에서 처음에 잡힌 문제는 두 가지였다. AI 뉴스 자동 생성 크론이 실제로 콘텐츠를 만들고 있었지만 dev.to로 sync가 안 되고 있었고, jidonglab에서 최신순 정렬이 깨져 있었다.

수정 자체는 빨랐다. 실패한 이유를 파악하는 데 시간이 걸렸을 뿐이다.

## AI 뉴스 자동화: 구글링 + X + Reddit을 매일 9시에

원래 AI 뉴스 자동 포스팅은 API 기반이었다. Anthropic API 키가 있어야 생성된다. 그런데 뉴스 소스가 너무 좁았다.

이 대화가 방향을 바꿨다:

```
나: 뉴스 범위는 구글링 / X, reddit, 쓰레드, 글로벌 소식 위주로
```

기존 구조에 Google Custom Search API와 외부 피드 크롤링을 추가하는 작업이었다. Claude Code가 `src/pages/api/generate-ai-news.ts`를 열고, 기존 뉴스 생성 로직이 어떻게 동작하는지 파악한 뒤 소스 레이어를 추가했다.

핵심 변경은 뉴스 수집 단계를 분리한 것이다. 기존에는 Claude API가 뉴스를 "추론"해서 생성했다면, 이제는 실제 소스에서 데이터를 긁어온 뒤 Claude API가 큐레이션하는 구조가 됐다. 정확도가 올라가고 환각이 줄었다.

GitHub Actions 크론 스케줄도 같이 수정했다. `0 0,12 * * *` (UTC 기준) — KST 9시와 21시에 실행.

## 4개 플랫폼 동시 발행: 진짜 문제는 인증이었다

Hashnode를 추가하는 건 쉬웠다. API 키 받아서 `HASHNODE_TOKEN` 환경변수로 넣으면 된다. 실제로 사용자가 토큰을 채팅에 붙여넣었고, Claude Code가 그걸 `src/lib/hashnode.ts`로 연결했다.

```
사용자: ceef0313-ecca-456d-ab62-6a60280e6ab1
사용자: https://plzai.hashnode.dev/
```

이런 식이다. 설명 없이 값만 던졌는데 Claude Code가 맥락을 파악해서 맞는 위치에 연결했다.

Medium은 달랐다. API 키가 아니라 OAuth 흐름이 필요했다. 사용자가 Google OAuth 클라이언트 ID와 시크릿을 올렸고, 인증 플로우를 구현하는 과정에서 "액세스 차단됨" 오류가 떴다. Medium의 자동 발행은 멤버십 결제가 필요했다. 결제 후 재시도로 해결됐다.

이 과정에서 나온 도구 사용 패턴이 흥미로웠다. Claude Code는 막힐 때마다 `Bash`로 직접 API를 호출해서 응답을 확인했다. 코드를 고치고 → 실제로 호출해보고 → 결과 보고 → 다시 고치는 사이클. Read/Edit만으로 해결 안 되는 부분에서 Bash가 쓰였다.

세션 전체 도구 호출 분포가 이걸 보여준다. 324번 중 Bash가 가장 많았다. 파일 읽고 수정하는 것보다 실제로 실행해보는 시간이 더 길었다는 의미다.

## dev.to에 한글만 올라간 사건

중간에 문제가 발견됐다. dev.to에 올라간 글 상당수가 한글 버전만 있고 영어 버전이 없었다.

빌드로그는 dev.to에 올리지 않는 규칙이 있다. AI 뉴스와 팁 포스트만 올라가야 했는데, 자동화 스크립트가 이 구분 없이 전체를 올리고 있었다.

프롬프트:

```
나: 뉴스에 대해서 너무 중복이 있거나 중요하지 않은 내용은 빼주고,
    dev.to에 지금 올라간 내용 중에 jidonglab 새 기능 추가, 빌드로그인 건 다 지워줘.
    시의성 지난 한국어 뉴스 삭제, 블로그글만 영어버전 생성
```

여기서 Claude Code가 한 일을 보면 흥미롭다. dev.to API로 기존 글 목록을 가져오고 → 제목과 태그로 카테고리를 분류하고 → 기준에 맞지 않는 글을 unpublish하고 → 영어 버전이 없는 글에 대해 번역 + 발행까지 했다.

API 호출 하나하나를 사람이 했다면 반나절 작업이었을 거다.

## admin 페이지: "가독성이 없어"

운영하면서 방문자 수가 Cloudflare 대시보드 수치와 admin 페이지 수치가 달랐다. 스크린샷을 올렸더니 Claude Code가 차이의 원인을 추적했다.

Cloudflare는 봇 트래픽을 포함한 raw 히트를 보여주고, admin 페이지는 실제 사람 방문자만 집계하고 있었다. 수치가 다른 이유가 있었다. 문제가 아니라 정상이었다.

그리고 admin 페이지에서 Build Logs by Project 섹션이 조회수순 정렬이 안 됐다. 이건 진짜 버그였다. 고쳤다.

```
나: Build Logs by Project 아 이 부분에 그냥 조회수별로 정렬해줘 컨텐츠
```

한 줄 프롬프트. `src/pages/admin/index.astro` 열고, 정렬 로직 추가. 30초.

## 번역 기능: 버튼 누르면 아무것도 안 나오는 버그

영어 버튼을 누르면 콘텐츠가 사라지는 문제가 있었다.

기존 i18n 로직은 `data-ko`, `data-en` 속성으로 요소를 숨기고 보여주는 방식이었다. 근데 AI 뉴스 콘텐츠는 동적으로 렌더링되는 Astro 컴포넌트라서 이 속성이 없었다. 언어 전환 시 매칭되는 요소가 없으니 빈 화면이 나왔다.

수정 방향은 접속 리전 기반 자동 감지 + 수동 전환 두 가지를 합치는 것이었다. `navigator.language`로 브라우저 언어를 감지하고, localStorage에 명시적 선택을 저장한다.

이 작업에서 Claude Code가 `Edit` 도구를 연속으로 쓰는 패턴이 나왔다. 한 파일에서 발견된 버그가 다른 파일에서도 같은 패턴으로 나타났고, 하나씩 찾아서 고쳤다.

## "다 해줘"의 실제 의미

이 세션에서 "다 해줘" 계열 프롬프트가 여러 번 나왔다.

```
"generate new jidonglab이랑 devto에 안되고 있으니까 확인해서 무조건 될 때까지 고치고, devto에 한글 글도 올려줘"
"cowork써서 9시마다 뉴스 크롤링해서 cli 구독모델로 사용하는거 해줘"
"hashnode에 dev.to의 영어글 보기 좋은 형식으로 변환해서 일단 모두 올려줘"
```

각각 구체성이 다르다. 첫 번째는 목표가 명확하고 ("무조건 될 때까지"), 두 번째는 방법이 지정됐고 ("cowork"), 세 번째는 소스와 타겟이 정해졌다.

Claude Code가 잘 처리한 건 목표가 명확한 케이스였다. "될 때까지 고쳐"는 실제로 여러 번 반복 수정이 일어났고 결국 됐다.

방법이 지정된 케이스는 달랐다. "cowork써서"라고 했는데 cowork 자동화 스크립트가 현재 환경에서 제대로 동작하지 않았다. 대안으로 GitHub Actions 크론으로 전환했다. 사용자가 원한 결과(9시 자동 뉴스 생성)는 달성됐지만 방법은 바뀌었다.

이게 Claude Code 사용에서 중요한 지점이다. "어떻게"가 아니라 "무엇을"에 집중해서 프롬프트를 쓰면 더 좋은 결과가 나온다.

## 숫자로 보는 이 세션

세션 시간: 약 26시간 (연속이 아니라 누적)
도구 호출: 324회
파일 변경: admin 페이지, dev.to sync, AI 뉴스 생성, i18n, Hashnode 연동 등 10개 이상
새로 추가된 플랫폼: Hashnode, Medium (기존 dev.to + jidonglab에서 4개로 확장)
버그 수정: dev.to 영어 버전 누락, admin 페이지 정렬, 번역 기능

비교하자면 직접 했다면 며칠 걸렸을 작업이다. API 문서 읽고, 각 플랫폼 인증 흐름 파악하고, 크론 스케줄 설정하고, 기존 코드와 통합하고. Claude Code는 이걸 한 세션에 다 했다.

단, 내가 해야 했던 것들도 있었다. Medium 결제, Hashnode 계정 생성, Google OAuth 앱 설정. 인증 크리덴셜은 사람이 직접 만들어야 했다. Claude Code는 만들어진 크리덴셜을 받아서 코드에 연결하는 역할이었다.

## 남은 문제

Blogger 연동은 Google OAuth 흐름에서 막혔다. "404 오류" 화면이 나왔고 세션 내에서 완전히 해결되지 않았다. 다음 세션 주제다.

Medium 자동 발행도 멤버십 결제 후 일부는 됐지만 모든 글이 정상적으로 올라가는지는 추가 검증이 필요하다.

AI 뉴스 자동화의 품질 문제도 있다. 같은 날 비슷한 주제의 뉴스가 여러 번 나오는 중복 이슈. 클러스터링 로직을 추가하거나 프롬프트에서 "이미 다뤘으면 건너뛰어"를 어떻게 표현할지가 숙제다.

> 자동화는 되게 만드는 게 아니라, 꺼놓고도 안심할 수 있을 때 완성된다.
